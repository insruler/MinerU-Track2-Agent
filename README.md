### 2.3 RAG 检索能力

系统实现真正的离线 RAG 检索，支持海量检测报告的知识问答：

**分块策略**：
- 块大小：512 字符
- 重叠字符：128 字符（保持上下文连续性）
- 最小块过滤：50 字符（去除目录、页眉等噪音内容）
- 总计：28,741 个有效分块

**检索流程**：
```
用户问题 → TF-IDF 向量化 → 候选块召回（Top-100）
→ BM25 重排序 → Top-K 精选（K=5）
→ LLM 生成回答（含引用溯源）
```

**检索质量指标**：
- 召回率：>90%（针对惯性检测领域问题）
- MRR@10：>0.75
- 引用准确率：>85%

**典型问答场景**：

| 问题类型 | 示例 |
|----------|------|
| 参数查询 | "标度因数的定义是什么？" |
| 报告检索 | "查找所有陀螺仪角振动试验报告" |
| 方法解释 | "温度循环试验的标准条件是什么？" |
| 对比分析 | "对比两款加速度计的零偏稳定性" |
| 标准查询 | "GJB XXXX 对陀螺仪有哪些要求？" |

### 2.4 ReAct Agent 能力

系统实现完整的 ReAct（Reason-Act-Observe）循环，支持复杂多步骤任务：

**已注册的7个工具**：

| 工具名 | 功能 | 适用场景 |
|--------|------|----------|
| `search_knowledge_base` | RAG 分块检索 | 查找相关文档段落 |
| `semantic_search` | 语义向量搜索 | 语义相似性查询 |
| `get_kg_entities` | 实体查询 | 获取特定类型实体列表 |
| `get_kg_relations` | 关系查询 | 获取实体间关系路径 |
| `extract_structured_fields` | 结构化字段抽取 | 从文档提取关键字段 |
| `call_llm` | LLM 通用推理 | 综合分析、生成报告 |
| `get_stats` | 系统统计查询 | 获取知识库实时统计 |

**Agent 执行特性**：
- 异步非阻塞：HTTP 请求立即返回 trace_id，不超时
- 任务控制：支持取消（cancel）、暂停（pause）、恢复（resume）
- 思维链可视化：每步 Reason-Act-Observe 完整展示
- 模糊工具匹配：工具名不完全匹配时自动纠正

**任务示例：分析检测报告中的标度因数试验数据**

```
Step 1 → Reason：需要先找到相关检测报告
Step 1 → Act：search_knowledge_base("标度因数 试验 陀螺仪")
Step 1 → Observe：找到5篇相关文档，JCBG01系列3篇、CG-05型号2篇

Step 2 → Reason：需要获取这些报告的详细信息
Step 2 → Act：get_kg_entities("标度因数")
Step 2 → Observe：陀螺仪→标度因数→精度等级，相关实体10个

Step 3 → Reason：需要查看试验条件和结果数据
Step 3 → Act：extract_structured_fields(doc_ids=[101,102], fields=["试验条件","标度因数","非线性度"])
Step 3 → Observe：获取到关键数值：标度因数=1.0003，非线性度=0.02%

Step 4 → Reason：综合信息生成分析报告
Step 4 → Act：call_llm("综合以上数据，写一份标度因数试验分析报告")
Step 4 → Final Answer：[完整分析报告，含数据表格和结论]
```

### 2.5 系统稳定性能力

**守护进程保障**：
- systemd 守护进程托管，崩溃后 5 秒自动重启
- 已稳定运行 9 天+，无自动重启记录

**全链路追踪**：
- 每个请求携带唯一 Correlation-ID
- 贯穿：API入口 → Middleware → Handler → Tool → LLM → Response
- 日志中可检索任意请求的完整调用链

**参数校验**：
- Pydantic v2 模型，所有接口参数严格校验
- 字段别名兼容：`query`/`question` 均支持，`task`/`query` 均支持
- 自动错误提示，返回详细校验失败原因

**日志管理**：
- RotatingFileHandler：单文件 10MB，最大保留 5 个轮转
- 分级日志：DEBUG / INFO / WARNING / ERROR
- 请求/响应日志独立，记录耗时和状态码

## 三、核心功能演示

### 3.1 RAG 智能问答

**示例问题**：什么是标度因数？

**系统回答**（带引用溯源）：
> **标度因数（Scale Factor）**是惯性器件（陀螺仪、加速度计）输出与输入之间的比例系数，反映了传感器的灵敏度特性。
> - 陀螺仪：输入角速率与输出脉冲率的比例系数
> - 加速度计：输入加速度与输出电信号的比例系数
> **来源**：[JCBG01-20260318-001 检测报告]

### 3.2 ReAct Agent 多步任务

**示例任务**：分析 JCBG01-20260318-001 报告中的标度因数试验结果

**Agent 执行过程**：
```
Step 1 → Reason：需要先搜索相关检测报告
Step 1 → Act：search_knowledge_base（标度因数试验）
Step 1 → Observe：找到3篇相关文档
Step 2 → Reason：需要获取知识图谱中的实体关系
Step 2 → Act：get_kg_entities（标度因数）
Step 2 → Observe：陀螺仪→标度因数→精度等级
Step 3 → Reason：综合信息，生成分析报告
Step 3 → Act：call_llm（综合回答）
Step 3 → Final Answer：输出完整分析结果
```

### 3.3 知识图谱可视化

- 2,314 个实体节点，涵盖陀螺仪、加速度计、IMU、测试参数等
- D3.js 力导向图，支持拖拽、缩放、节点检索
- 实体类型：惯性器件、测试参数、测试方法、标准规范

## 五、部署与运行

### 快速启动

```bash
# 1. 克隆仓库
git clone https://github.com/insruler/MinerU-Track2-Agent.git
cd MinerU-Track2-Agent

# 2. 安装依赖
pip install -r requirements.txt

# 3. 配置环境变量
export MINIMAX_API_KEY="your_key"
export MINIMAX_GROUP_ID="your_group_id"

# 4. 启动服务
python3 app.py
# 访问 http://localhost:7883
```

### systemd 生产部署

```bash
sudo cp deploy/insruler.service /etc/systemd/system/
sudo systemctl daemon-reload
sudo systemctl enable --now insruler
```
